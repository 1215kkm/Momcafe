**PM**: 방금 CEO가 큰 결정 내렸어요. "한 동네 말고 누구나 카페처럼 만들 수 있게." 이건 정체성 자체가 바뀌는 거. 4번째 안은 어떻게 되는 거지

**CEO보좌**: 일단 *카페별 옵션*으로. 보안은 모듈 정의 라운드에 끼어주세요.

## 회의 2. "오늘의 결정" 메커니즘의 운명

- **주제**: 4번째 안 메커니즘을 옵션 모듈만으로 둘지, 플랫폼 레벨까지 격상할지
- **일시**: 2026-05-21 14:45
- **참석자**: PM · 아뱅 · UI · UX · 마케터 · QA · CEO보좌

**PM**: 이게 가장 큰 분기. 4번째 안의 핵심을 어디에 놓느냐.

**아뱅**: 저는 *둘 다*에 놓자고 봐요. 옵션 모듈만 하면 사용자가 첫 화면 들어왔을 때 *왜 이 앱이 다른지* 안 보임. 그냥 네이버 카페 같음.

**UI**: 첫 화면에 매일 19시 *한 모임을 클로즈업*해서 "가입할지 보류할지" 30초 결정하는 거요. 디자인 단계에서 *드라마틱한 한 장*이 가능.

**UX**: 근데 그러면 D01 발견 홈의 *카드 리스트*와 충돌. 사용자가 이미 카드 형식으로 보여달라고 명시했음.

**아뱅**: 충돌 아님. 카드 리스트 *위에 1장* 얹는 거. 풀스크린이 아니라 *최상단 한 장* + 그 아래 카드 리스트.

**마케터**: 19시는 식사 시간이라 푸시 열어볼 확률 낮을 수도. 7시(아침) vs 21시(아이 재운 뒤) 중 21시가 더 강할 수도.

**PM**: 그건 마케팅 라운드에서 A/B. 일단 *플랫폼 + 모듈 두 층*으로 가는 게 강팀 다수안.

**QA**: 두 층이면 4번째 안 메커니즘이 *카페 안*에서도 작동해야 하는데, 카페 안 결정 vs 플랫폼 가입 결정이 *어떻게 다른가*를 사용자가 헷갈리지 않게 *주제*를 명확히 갈라야 해요.

**아뱅**: 플랫폼 = "어느 모임에 가입할지". 카페 안 = "이 모임 안에서 무엇을 할지". 같은 메커니즘, 다른 결정. 헷갈릴 일 없음.

**CEO보좌**: 추천안으로 Q1 (b) 정리해서 CEO에게 올립니다.

## 회의 3. 7개 특수 게시판 (아뱅 발산 라운드)

- **주제**: 모임 운영자가 켤 수 있는 특수 게시판 카탈로그 도출
- **일시**: 2026-05-21 15:10
- **참석자**: 아뱅 · UX · 보안 · 개발 · 마케터 · QA · CEO보좌

**아뱅**: 일단 7개 다 던집니다. 1.오늘의결정 2.이름회복실 3.내일의나에게 4.심박수 5.고백 6.예측시장 7.교환.

**UX**: 이거 다 만들면 모임 안 IA가 폭발해요. 한 카페에 게시판 10개씩 있으면 사용자 인지 부하…

**아뱅**: 그래서 *옵션*. 운영자가 켜야 작동. 켜는 동기는 *처방* 형식으로. "당신 모임이 죽고 있으니 이걸 켜세요."

**보안**: 6번 예측시장은 *현금 베팅 절대 금지*. 명예 포인트로만. 안 그러면 사행성 규제 직격.

**아뱅**: 처음부터 명예 포인트로 설계함. 보안 우려 반영.

**개발**: 5번 고백 게시판의 *완전 익명(본인도 추적 불가)* 구현은 진짜 어려워요. 키 폐기 후 복구 불가 = 백업 복원도 불가. 데이터 손실 리스크.

**아뱅**: 그게 *기능*임. 추적 가능하면 익명이 아님. 백업 복원 안 됨이 *기능 약속*.

**보안**: 단, *위법 콘텐츠(아동·테러 등) 수사 협조 요청*이 들어왔을 때 *복원 불가*가 법적 리스크. 한국 통신비밀보호법·정보통신망법 확인 필요.

**PM**: 이건 보안 라운드에서 본격 검토. MVP에 고백 게시판이 들어갈지부터 CEO에게 묻자.

**마케터**: 4번 심박수는 *플랫폼 자체의 마케팅 메시지*가 됨. "당신이 가입한 카페가 살아있는지 보세요" — 광고 카피 그대로 사용 가능.

**QA**: 4번째 안 3개 + 심박수 4개가 *MVP 권장*. 1, 2, 3은 이미 검증된 메커니즘이고 4는 플랫폼 차별점. 5, 6, 7은 v1.5.

**CEO보좌**: Q2 추천안 (a)로 정리.

## 회의 4. 모임 카드 레이아웃

- **주제**: 사용자 명시 카드 형식(좌 로고, 우 최근글 2 + 인기글 2)을 360px에서 어떻게 구현할까
- **일시**: 2026-05-21 15:35
- **참석자**: UI · UX · 아뱅 · QA · 개발 · PM

**UI**: CEO가 명시했어요. *좌측 로고, 우측에 최근글 2 + 인기글 2 위아래로*. 근데 360px 모바일에서 텍스트 4줄이 들어가면 한 줄에 글자 18자가 한계. 제목 잘림 우려.

**UX**: 그러면 펼치기형으로 가야…

**UI**: 잠깐. CEO가 *명시*했어요. 우리가 함부로 안 바꿉니다. 두 안 다 와이어로 그려서 CEO가 선택하게 하죠.

**아뱅**: CEO 명시 안을 *지키되* 비주얼 트릭으로 풀면? 제목을 그라데이션 페이드아웃으로 자르고 글자 작게(13px) 표시. 인지 부하는 *흥미로운 내용 4개 미리보기*라는 가치로 상쇄.

**UI**: 좋아요. A안(CEO 명시) 베이스, 360px 풀스크린 와이어로 *살아있게 보이는지* 검증하고, 안 되면 B안(펼치기) 옵션 제시.

**QA**: 카드에 흥미 유발하려면 *최근 2 + 인기 2*가 진짜 흥미로운 글이어야 함. 알고리즘이 *시선 잡는 한 줄*을 잘 골라야 함.

**개발**: 인기 알고리즘은 *24시간 댓글수 + 좋아요 가중치*로 일단 단순하게. 머신러닝은 데이터 쌓인 뒤.

**PM**: UI 와이어 단계에서 A안 풀그림 + B안 옵션. CEO가 와이어 보고 결정.

## 회의 5. 운영자 권력 구조

- **주제**: 운영자·부운영자·게시판 담당 위계와 강제 탈퇴 권한, 헌법 P4와의 충돌 해소
- **일시**: 2026-05-21 16:00
- **참석자**: PM · 아뱅 · CEO보좌 · UX · UI · 개발 · QA · 보안

**PM**: 방금 CEO 지시 들어왔어요. *운영자·부운영자·게시판 담당 + 강제 탈퇴 권한*. "막강한 권력자여야 한다"고 명시.

**아뱅**: 잠깐. 헌법 P4 "댓글권은 보상이지 권력이 아니다"와 정면 충돌 아닌가요? 우리는 운영자 폭정을 *반대*하는 플랫폼이었는데.

**CEO보좌**: P4는 *멤버의 발언권*에 관한 규칙이고, *운영자 권한*에 관한 규칙은 따로. 모순 아닙니다. 다만 *견제 장치*가 함께 가야 함.

**UX**: 견제 장치 후보: 1) 아이디어 게시판 자동 채택 2) 모든 권력 행사 공개 로그 3) 고백 게시판 4) 사망 경고. 네 개로 운영자 폭정을 견제.

**아뱅**: 좋아요. "*막강한 권력 + 공개된 권력*"이 컨셉. 운영자는 강하지만 *모든 강함이 멤버에게 보인다*. 비공개 정치 불가.

**보안**: 강제 탈퇴 시 *사유 1줄 필수*. 그리고 *공개 로그*. 누가 누구를 왜 강퇴했는지 보임. 운영자가 사유 거짓말하면 멤버 신뢰 자동 손실. 셀프 견제.

**개발**: 부운영자 권한이 *체크박스 5종*이라고 했는데 더 늘어날 수도. UI에서 권한 *그룹 프리셋*("일반 부운영자"·"강퇴 전담"·"게시판 전담")을 만들어 두면 편함.

**UI**: 채택. M40 화면에서 *프리셋 3종 + 커스텀*.

**QA**: 강퇴 당한 멤버가 *복수 계정으로 재가입*하면? *기기/이메일/전화 다중 검사*가 필요할 수도. 보안 라운드에서.

**보안**: 메모. v1.5 보안 모듈에 *재가입 차단 기제* 검토.

**PM**: 추천: MVP에 운영자·부운영자·게시판 담당·강퇴 *다 탑재*. 견제 장치 4종 동시 작동. CEO에게 *권한 견제 정도* 결정 받아옴.

## 회의 8. 세분화 모임 생성 + 홍보 4단계 로드맵

- **주제**: 모임 세분화 가능 여부 + 광고비 0 마케팅 로드맵
- **일시**: 2026-05-21 16:40
- **참석자**: PM · 아뱅 · UX · 마케터 · 보안 · 개발 · QA · CEO보좌

**PM**: CEO가 두 가지 물었어요. 첫째, 모임이 *세분화*되는 건 맞느냐 ("3~7세 미만 아이들 엄마 모임" 같은 좁은 단위). 둘째, *홍보 전략*은 어떻게.

**아뱅**: 첫째는 *그게 핵심*이에요. 네이버 카페가 죽는 이유 중 하나가 *너무 넓은 범위*("강서구 맘카페" 30만 명)예요. 우리는 정반대. 좁을수록 살아남음. "강서구 7세 비건 엄마"가 가입자 30명이라도 *심박수 100*이면 살아있는 거.

**UX**: 그래서 *모임 만들기 5단계*(M01~M05)에서 *카테고리 + 세부 태그*를 다 받음. 카테고리만 받으면 또 넓어짐. 태그가 핵심.

**마케터**: 좋습니다. 이제 홍보. 광고 인벤토리 0이라는 헌법 때문에 *내부 광고*는 못 팔지만, *외부 채널 마케팅*은 강팀 책임이에요. 안 7개 가져왔어요.

**마케터** (안 7개): ①결핍 폭격 광고 ②죽은 카페 백서 PR ③이름 회복실 1시간 무료 체험 ④모범 운영자 인플루언서화 ⑤B2B 역방향 마케팅 ⑥데이터 백서 연간 발행 ⑦사망 모임 장례식 라이브

**아뱅**: 7개 중 안 ①②⑤는 *광고가 아닌 자산 활용*. ③④⑥⑦은 *광고 없이도 외부 확산*. 헌법 P5(광고 아님)를 *마케팅에서도* 지킬 수 있다는 게 강팀의 차별점.

**UX**: 그러면 카페 홈(C01)에 *동시 영상* 영역이 항상 보임. 게시판·모듈과 별개 영역. 새 화면 ID: **M21** 동시 영상 시간 설정(운영자), **C07** 동시 영상 풀스크린 보기, **B07** 동시 영상 게시판 (모임이 옵션으로 켤 수 있는 별도 게시판 — 운영자 선택).

**UI**: 디자인은 셋로그 그대로 — *세로 그리드*. 한 화면에 4~9개 영상 동시 재생. 자동 mute, 탭하면 소리. *2~10초 짧은 영상* 강제.

**개발**: 기술 구현 — secret-ad의 인프라에 WebRTC 모듈 추가 + 시그널링 서버. 약 3주 작업. 옵션 B (메타만 서버) — 영상은 폰 ↔ 폰 P2P, 우리는 *썸네일·시각·시간 윈도우* 메타만 저장. NestJS에 시그널링 모듈 한 개.

**보안**: 세 가지 우려.
1. *시간·위치 메타*에서 *추적 가능성*. 익명화 + 정밀도 다운(시간은 ±5분 윈도우, 위치는 모임 동네 단위까지만).
2. *미성년자 보호* — 셋로그 모듈 가입자에게 가입 시 *연령 인증* 필요. 18세 미만에게는 *자동 모자이크*.
3. *P2P 전송 암호화* 필수. WebRTC는 SRTP 기본 암호화 — OK.

**QA**: 시간대 차이 처리. *시간 약속 = 모임 동네 시간 기준*. 예: 운영자가 "오후 3시" 설정 → 강서구 모임이면 *강서구 현지 시간 오후 3시*. 다른 지역 멤버도 *같은 동네 시간*에 찍어야 함. *동네 단위 동기성* 강조.

**아뱅**: QA 좋아. 그게 셋로그의 정신. *지금 이 동네 풍경*을 *같은 동네 사람만* 본다는 것.

**마케터**: 셋로그 영상이 *공유 카드*로 변환되면 SNS 폭발 가능. 단 본인 동의 필수. 또 *셋로그 모임 차별점* — "내 동네에서 지금 무슨 일이 일어나고 있나" — 우리만의 콘텐츠 형식.

**보안**: 셋로그 영상의 *공유 카드 변환*은 *기본 OFF*. 본인이 명시 동의 후에만. 다른 사람 영상은 *절대 외부 공유 불가*.

**UX**: 카페 홈에 [📹 지금 모임 동시 영상] 영역이 *상단*에 보임. 시간 윈도우 안에 있으면 *촬영 가능* 버튼, 윈도우 밖이면 *지난 영상 모음 보기*.

**개발**: 영상이 폰에만 저장 → *폰 분실·교체 시 영상 사라짐*. 그것도 *기능 약속*. "그 순간 함께한 사람만 영원히 기억한다."

**아뱅**: 좋아요. *영상의 사라짐*이 셋로그의 정신. 인스타 스토리·스냅챗과 같은 층위. 헌법에 *영상 시간 휘발성 (P3+α)* 추가 권장.

**PM**: 운영자 권한 — 시간 윈도우 설정·해제 (M21). 한 모임이 *주 N회* 시간 윈도우 설정 가능.

**CEO보좌**: 정리 — 셋로그는 *플랫폼 기본 기능*(모든 모임 자동 사용 가능, 운영자가 끌 수 있음). 영상은 폰 저장 + 메타만 서버. 헌법에 *영상 시간 휘발성* 추가. 결정 분기 Q12 추가 (시간 단위 옵션) — 강팀 추천 (a) 모임 동네 시간.

**PM**: 구현은 *코드 작업 단계*에서. 지금은 기획 박제만.
